## Why

当前 CLI 已支持 UI 动作、断言、截图、手势以及 bridge-backed 小程序动作，但还不能对这些流程触发的网络请求进行观察和控制。这导致我们很难验证某个用户操作是否真的发出了预期请求，也很难以可重复、可确定的方式复现后端失败、超时或 mock 数据等场景。

## What Changes

- 新增结构化的网络观测能力，使计划在执行过程中可以监听出站请求以及匹配到的响应。
- 新增可过滤的网络断言能力，支持回答常见测试问题，例如“某个动作之后是否发出了任意请求”以及“是否发出了符合 URL、Method、params、headers、body 等条件的特定请求”。
- 新增结构化的请求拦截控制能力，使计划可以对匹配请求执行透传、失败注入、延迟注入或 mock 响应返回，而不依赖外部后端预置环境。
- 为监听器和拦截规则增加作用域与生命周期控制，确保同一计划多次执行时网络捕获行为仍然稳定、可预测。
- 扩展机器可读的步骤结果和运行产物，使其包含请求匹配证据与拦截摘要，便于调试与 CI 消费。
- 在用户初始诉求之外补充常用能力，包括等待某个匹配请求、按次数断言请求、断言多个请求的发生顺序，以及在可用时检查响应状态码或响应体。
- 优化仓库内置 product-use skill 对执行命令的指导，避免 agent 在只需要读取落盘产物时默认附带 `--json`，从而减少大体积结构化结果回传带来的 token 消耗。

## Capabilities

### New Capabilities
- `miniapp-network-observation`：在结构化计划中观测出站网络流量，按条件过滤请求与响应，等待匹配流量，并对请求证据进行断言。
- `miniapp-network-interception`：拦截匹配请求，并对其施加确定性的处理结果，例如继续放行、强制失败、人工延迟或直接返回 mock 响应。

### Modified Capabilities
- `miniapp-cli-test-execution`：扩展执行与产物语义，使网络事件、请求断言结果和拦截结果能够在步骤结果与运行落盘产物中保持一致表达。
- `cli-agent-skills`：补充 product-use skill 的执行指导，要求 agent 先依据 CLI 结论判断验证目标是否满足，仅在需要进一步证据时再按需读取落盘产物，并将 `--json` 视为按需开启的结构化 stdout 模式，而不是默认执行参数。

## Impact

- 为新的网络相关 step 类型及其过滤条件补充计划 schema 与校验逻辑
- 扩展 CLI 执行引擎与托管 Python 运行时桥接层，以捕获并控制小程序网络流量
- 扩展结构化结果载荷与 `.minium-cli/runs` 下的运行产物，以记录网络证据
- 更新产品文档，例如 `README.md`、`docs/API_REFERENCE.md` 以及随仓库发布的 skill 指南
- 更新 repository-managed skill 的执行参考，明确 `--json` 的适用场景与产物优先的分析路径
- 为请求观测、mock、失败注入等场景补充示例计划和示例小程序覆盖
