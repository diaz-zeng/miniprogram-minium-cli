## Context

当前仓库只有一个 Node.js CLI 占位入口，而参考仓库 `https://github.com/diaz-zeng/miniprogram-minium-mcp` 已经通过 spec-driven 方式验证了小程序自动化与测试的核心能力边界，包括：

- `miniapp-acceptance-session`：运行时准备、会话创建、状态读取、关闭与超时清理
- `miniapp-acceptance-actions`：查询、点击、输入、等待、断言与失败取证
- `miniapp-acceptance-gestures`：两指以内的复合手势与触点状态维护

这次变更的难点不在“能不能驱动 Minium”，而在“如何把 MCP 风格的工具调用接口重组为 AI-first CLI 执行层”。新的 CLI 需要同时满足几个约束：

- 用户入口应当是命令行，而不是 MCP tool 或 stdio server。
- 自动化与测试能力要尽量复用 `https://github.com/diaz-zeng/miniprogram-minium-mcp` 已验证的领域模型和 requirement 语义，避免重新发明一套不兼容的能力边界。
- AI-first 不能等价于“执行时每一步都依赖大模型在线决定”，否则稳定性、可重放性和调试性都会很差。
- 规划应由上层 agent 或外部工具完成，CLI 只负责结构化计划的消费、执行与结果输出。

主要相关方包括：

- 通过 agent 驱动自动化与测试的开发者、测试同学
- 维护本地执行环境的使用者，他们需要看懂依赖、错误和产物
- 后续实现者，他们需要在不引入 MCP 协议包袱的前提下复用已有能力

## Goals / Non-Goals

**Goals:**

- 提供一个 AI-first CLI 执行层，支持消费上层 agent 生成的结构化测试计划。
- 复用 `https://github.com/diaz-zeng/miniprogram-minium-mcp` 中与 MCP 无关的运行时、领域服务和错误语义，沉淀为 CLI 可调用的内部模块。
- 覆盖小程序自动化主链路：会话、查询、点击、输入、等待、断言、截图、多触点手势、结果汇总。
- 让上层规划和 CLI 执行解耦，形成“外部生成计划、CLI 执行、可回放”的测试模型。
- 输出适合 CLI 使用的执行摘要、失败证据、退出码和产物目录。

**Non-Goals:**

- 不实现 MCP Server、tool 注册、stdio 协议适配或任何面向 MCP 客户端的接口。
- 不在首期做任意脚本执行器、任意 JS 注入器或底层 Minium API 全量映射。
- 不在首期做远程分布式执行、多租户任务调度或云端托管。
- 不把 AI 编排设计成黑盒代理循环；首期优先保证执行计划可解释、可审计、可复现。
- 不在 CLI 内实现自然语言 planner。

## Decisions

### 1. 采用“Node CLI 壳层 + Python Minium 执行内核”的双层实现

设计：

- 保留当前仓库的 Node.js 包入口与 `bin` 分发方式，CLI 主入口继续由 Node 提供。
- 新增 Python 执行内核，承载 Minium 适配、会话管理、动作、断言、手势和取证等核心运行时能力。
- Node CLI 负责命令解析、计划输入处理、结果渲染和进程编排；Python 内核负责真正的自动化执行。
- Node 与 Python 之间采用本地子进程 + JSON 输入输出的边界，而不是 MCP 或自定义长连接协议。

原因：

- 当前仓库已经是 Node CLI 形态，保留 Node 壳层可以减少发布和用户入口变化。
- `https://github.com/diaz-zeng/miniprogram-minium-mcp` 的核心价值主要沉淀在 Python + Minium 这一侧，直接迁移或复用该层比在 Node 中重写一遍运行时更稳。
- 用 JSON 作为进程间契约，可以兼顾外部 agent 生成结果的可序列化与执行层的稳定性。

备选方案：

- 方案 A：整体迁移到纯 Python CLI。
  - 优点：与参考仓库技术栈一致，复用成本最低。
  - 缺点：偏离当前仓库形态，也会弱化 Node 生态下的 CLI 分发体验。
- 方案 B：全部重写为纯 Node + Minium 封装。
  - 优点：技术栈统一。
  - 缺点：需要重做一整套运行时与错误模型，风险明显更高。

结论：采用 Node 作为产品外壳、Python 作为执行内核的双层架构。

### 2. 用“External Planner -> Plan -> Executor”替换参考仓库中的 MCP tool 层

设计：

- 将参考项目中的 `server/tools` 职责替换为“外部 planner + CLI executor”模型：
  - `external planner`：由 agent 或外部工具把测试意图转为标准执行计划
  - `executor`：CLI 消费执行计划，调用 Python 内核完成会话、动作、断言和取证
- 执行计划采用稳定 JSON 结构，包含：
  - 项目与环境配置
  - 会话初始化参数
  - 顺序步骤列表
  - 断言与失败策略
  - 产物输出位置
- CLI 不提供内置自然语言入口，只消费结构化计划，不直接承担规划职责。

原因：

- MCP tool 层面向“外部工具调用”，CLI 则更适合“外部生成计划，再执行计划”的模式。
- 计划对象是 AI-first CLI 的关键中间产物，它让自然语言能力具备可审查、可缓存、可重放特性。
- 这能避免执行期过度依赖大模型，降低网络波动或模型漂移对测试稳定性的影响。

备选方案：

- 方案 A：AI 直接循环调用本地动作函数，不生成中间计划。
  - 优点：交互更像 Agent。
  - 缺点：难以复现、难以调试、难以回放。
- 方案 B：完全不做自然语言入口，只保留结构化命令。
  - 优点：边界清晰，更适合 agent 驱动。
  - 缺点：需要上层系统提供规划能力。

结论：以计划文件或内联 JSON 作为外部 agent 与执行层之间的稳定边界。

### 3. 领域分层复用参考仓库能力，但把协议语义改为 CLI 任务语义

设计：

- 内核继续保持类似 `domain -> adapters -> support` 的分层：
  - `domain/`：会话、动作、断言、手势、测试执行服务
  - `adapters/minium/`：对 Minium 的薄适配
  - `support/`：配置、日志、产物目录、国际化、进程辅助
- 删除或不引入参考项目中的 `server/` MCP 注册层。
- 在领域层新增 `test_execution` 聚合服务，把原来散落在多个 tool 调用之间的测试链路整合为一次 CLI 运行任务。
- 对外语义从“调用一个工具”变成“执行一个步骤”，但步骤本身仍沿用 `session`、`actions`、`gestures` 的 requirement 边界。

原因：

- 参考项目最有价值的部分不是 MCP 暴露方式，而是已经被 OpenSpec 固化的分层和错误模型。
- 通过保留领域分层，我们能最大化复用现有能力，同时避免 CLI 层直接耦合底层 Minium 细节。
- 新增 `test_execution` 聚合层，可以把多次动作组织为一次测试运行，而不是把 CLI 退化成“本地版 tool 调用器”。

备选方案：

- 方案 A：CLI 直接调用 Minium，不保留领域层。
  - 优点：实现快。
  - 缺点：后续动作、断言、手势、取证会高度耦合。
- 方案 B：原样照搬 `minium-mcp` 目录，仅删除 server。
  - 优点：迁移简单。
  - 缺点：缺少面向 CLI 任务执行的聚合抽象。

结论：保留领域分层，增加执行聚合层，删去 MCP 协议层。

### 4. AI-first 入口分为“文件计划执行”和“内联 JSON 计划执行”两种模式

设计：

- CLI 提供至少两类主入口：
  - `exec --plan <file>`：直接执行已有计划文件
  - `exec --plan-json <json>`：直接执行外部 agent 生成的内联 JSON 计划
- 计划需要是显式、可保存、可 diff 的 JSON 文档，便于用户审查和二次编辑。
- 计划中的步骤类型首期围绕以下能力展开：
  - `session.start`
  - `page.read`
  - `element.query`
  - `element.click`
  - `element.input`
  - `wait.for`
  - `assert.pagePath`
  - `assert.elementText`
  - `assert.elementVisible`
  - `gesture.touchStart/touchMove/touchTap/touchEnd`
  - `artifact.screenshot`
  - `session.close`

原因：

- AI-first 的价值在于让上层 agent 规划更灵活，同时保持 CLI 执行边界稳定。
- 两种执行入口并存，可以同时满足“我已有计划文件”和“agent 刚生成一份内联计划需要立即执行”这两类场景。
- 统一步骤类型也让后续 specs 可以直接映射到 plan schema 和执行器实现。

备选方案：

- 方案 A：只有计划文件入口。
  - 优点：最可控。
  - 缺点：agent 需要先落盘才能执行。
- 方案 B：只有内联 JSON 入口。
  - 优点：最适合 agent 即时调用。
  - 缺点：人类手动审查和回放体验较弱。

结论：文件计划负责可审查与可回放，内联 JSON 负责 agent 即时执行，两者共存。

### 5. 测试结果采用“结构化运行摘要 + 产物目录 + 明确退出码”

设计：

- 每次 CLI 运行都生成一个 `run summary`，至少包含：
  - 运行 ID
  - 项目路径
  - 步骤总数、成功数、失败数、跳过数
  - 失败步骤摘要
  - 截图与证据路径
  - 总耗时
- 退出码约定为：
  - `0`：全部成功
  - `1`：断言失败或动作失败
  - `2`：环境或配置错误
  - `3`：计划解析或参数错误
  - `4`：内部未预期错误
- 产物目录采用按运行 ID 分层的本地目录结构，便于和参考仓库现有截图/证据管理方式对齐。

原因：

- CLI 的一等输出不应该是随意打印日志，而应该是可读且可被上层脚本消费的运行结果。
- 明确的退出码是 CLI 与脚本、CI 集成的基础。
- 产物与摘要分离，可以兼顾终端可读性与故障排查深度。

备选方案：

- 方案 A：只打印控制台日志，不输出结构化摘要。
  - 优点：实现简单。
  - 缺点：难以自动化消费。
- 方案 B：只输出 JSON，不做人类可读摘要。
  - 优点：机器友好。
  - 缺点：终端体验较差。

结论：同时提供终端摘要与结构化结果文件，并用退出码表达最终状态。

### 6. 会话与手势模型沿用参考 specs，但执行粒度升级为“步骤级状态机”

设计：

- 保留 `session_id`、当前页面、最近截图、最近失败摘要等核心上下文。
- 保留 `active_pointers` 与 `latest_gesture_event`，继续支持两指以内的复合手势。
- 执行器按步骤顺序推进，步骤之间显式共享同一会话上下文。
- 若计划中未显式关闭会话，执行器在运行结束时统一做资源清理，并在摘要中标记自动清理结果。

原因：

- 参考项目已经证明，会话化和触点状态化是小程序复杂交互稳定执行的关键。
- CLI 场景下，步骤顺序由计划决定，因此把这些状态绑定到“本次运行上下文”比暴露给外部更自然。
- 自动兜底清理可以减少 CLI 用户因为漏写关闭动作而污染本地环境。

备选方案：

- 方案 A：每个步骤无状态执行。
  - 优点：模型简单。
  - 缺点：无法支持复杂交互和稳定上下文。
- 方案 B：完全隐藏会话概念，不允许计划显式操作。
  - 优点：表面更简单。
  - 缺点：失去灵活性，也不利于后续扩展。

结论：继续沿用会话与手势状态模型，但把它们封装在一次运行任务内部。

### 7. 环境配置沿用本地优先策略，并为 AI 规划与执行分别提供配置面

设计：

- 执行侧配置继续围绕本地开发环境：
  - 小程序项目路径
  - 微信开发者工具 CLI 路径
  - 自动化端口
  - 产物目录
  - 默认超时
- 上层 AI 规划侧单独维护模型与 prompt 配置，但不让这些配置污染执行计划中的领域语义。
- 外部生成的 plan 在进入 CLI 执行前必须把关键执行参数固化下来。

原因：

- 执行配置与 AI 配置属于不同生命周期，不拆开会导致计划不可复现。
- 关键执行参数写入 plan 后，即使后续默认配置变化，也不会影响历史计划重放。

备选方案：

- 方案 A：所有配置都只放环境变量，不进 plan。
  - 优点：简单。
  - 缺点：可复现性差。
- 方案 B：把所有配置都塞进上层 prompt。
  - 优点：对上层 agent 看起来更统一。
  - 缺点：执行边界模糊，难以验证。

结论：执行配置、AI 配置、计划内容三者分层管理。

### 8. Python 运行时采用 `uv` 托管模式，启动时懒加载并支持显式预热

设计：

- CLI 在首次真正需要执行 Python 内核时，自动在自身缓存目录中准备受控的 `uv`、托管 Python 与项目环境。
- 托管 Python 版本 MUST 为 3.11 或更高版本，默认请求 3.14，但允许在 3.14 不可用时回退到任意 `>= 3.11` 的托管版本。
- 运行时准备采用懒加载模式：`run` 或 `exec --plan` 首次进入执行阶段时才触发环境准备，而不是在 `npm install` 阶段预先创建。
- CLI 额外提供显式预热命令，用于提前下载 `uv`、托管 Python 和项目环境，以减少首次真实执行时的冷启动成本。
- 托管运行时目录应当位于用户级缓存路径下，并按操作系统使用合适位置，例如 macOS/Linux 使用用户缓存目录，Windows 使用 `%LOCALAPPDATA%`。
- Node CLI 仅在当前进程及其拉起的 `uv` / Python 子进程作用域内注入运行时相关环境变量，不修改用户全局 `PATH`、全局 `python`、全局 `pip` 或 shell 配置文件。
- 清理托管运行时时，应只影响 CLI 自己的缓存目录，不影响用户机器上其他 Python 项目或系统解释器。

原因：

- 这个 CLI 的目标形态是 npm 可安装、Python 对用户透明；如果要求用户自行管理 Python，会显著增加上手门槛。
- `uv --managed-python` 可以像参考仓库一样摆脱对用户本地 Python 的依赖，更符合“安装 npm CLI 后即可运行”的目标体验。
- 以 Python 3.11 作为最低基线，可以收敛过旧解释器带来的兼容负担；同时优先选择 3.14，有利于未来保持更一致的运行时行为。
- 自动化与测试真正依赖的是 Minium 和本地开发者工具环境，而不是让用户学习 Python 虚拟环境管理。
- 懒加载可以避免把重型环境准备塞进 `npm install`，而显式预热命令又能兼顾首次冷启动体验。
- 私有托管模式可以降低版本冲突风险，也更容易做跨平台发布与问题排查。

备选方案：

- 方案 A：要求用户预先安装和配置全局 Python 环境，再由 CLI 用本地 Python 创建 `venv`。
  - 优点：实现相对简单。
  - 缺点：仍然依赖用户本地 Python，不符合“Python 对用户透明”的目标。
- 方案 B：在 `npm install` 时就自动下载 Python 并创建环境。
  - 优点：首次执行更快。
  - 缺点：安装期副作用重，失败面更广，也不符合 npm 用户对安装阶段的预期。

结论：采用 `uv` 托管 Python 的私有运行时，默认在启动执行时懒加载，并额外提供显式预热命令。

## Risks / Trade-offs

- [双语言栈会增加工程复杂度] → 通过清晰的 JSON 契约限制 Node 与 Python 的耦合面，并优先复用参考仓库已验证的 Python 内核。
- [`uv` 托管运行时会带来明显冷启动成本] → 默认采用懒加载并复用缓存，同时提供显式 `prepare-runtime` 预热命令降低首次真实执行等待时间。
- [外部 agent 生成的计划可能不稳定] → 强制输入必须满足标准 plan schema，并支持落盘审查、编辑与重放。
- [复用参考项目时容易把 MCP 语义残留进 CLI] → 在实现上显式删除 `server/tools` 风格入口，只保留领域服务和适配层。
- [执行步骤过细会让计划冗长] → 首期接受这种显式性，以换取可解释性；后续再评估是否增加组合步骤模板。
- [本地环境依赖仍然脆弱] → 延续参考项目的环境错误模型，把依赖缺失和开发者工具问题明确分类到 `ENVIRONMENT_ERROR`。
- [Node 与 Python 之间的子进程边界可能影响性能] → 首期以稳定性和可复用性优先，后续如果出现瓶颈，再评估长驻内核进程或批处理执行模式。

## Migration Plan

1. 先定义 plan schema 与 CLI 命令面，确定文件计划入口和内联 JSON 执行入口。
2. 从 `https://github.com/diaz-zeng/miniprogram-minium-mcp` 对应实现中抽取与 MCP 无关的领域层、适配层和 support 层，迁移到本仓库的 Python 内核。
3. 在 Python 内核上新增 `test_execution` 聚合服务，使其能按计划顺序执行 `session/actions/gestures/assertions`。
4. 在 Node 侧实现 executor、结果渲染和本地进程编排。
5. 补齐 specs，让 `ai-first-cli-orchestration`、`miniapp-acceptance-session`、`miniapp-acceptance-actions`、`miniapp-acceptance-gestures`、`miniapp-cli-test-execution` 分别落地。
6. 最后再补 tasks，把实现拆成可提交的小步工作。

回滚策略：

- 该变更为新增能力，不替换现有线上系统；若实现中途发现双语言方案不可控，可先只保留结构化 plan 与 Node CLI 壳层，暂缓接入 Python 内核。
- 若上层 agent 规划能力尚不稳定，可先使用文件计划模式，再逐步引入内联 JSON 执行模式。

## Open Questions

- Python 内核是直接在本仓库内嵌，还是以可复用子包形式从参考仓库侧抽取公共模块？
- 计划 schema 首期是否要支持条件分支、重试和循环，还是仅支持顺序步骤以收敛复杂度？
- 结果输出除了终端摘要和 JSON 文件，是否还需要首期支持 JUnit 等 CI 友好格式？
