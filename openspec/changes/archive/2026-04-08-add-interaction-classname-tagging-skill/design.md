## Context

本次变更要新增一个仓库维护的开发期 skill，用来指导 agent 在编写或修改小程序页面、组件时，为点击、输入、切换、选择、提交等可交互元素显式补充专用的 `className` 标识。当前仓库已经有一个面向 `miniprogram-minium-cli` 产品使用的 skill，但还没有针对“开发期可测性打标”这一协作动作的专用 skill。

这个设计需要同时满足几类约束：

- skill 必须遵循仓库现有的标准 skill 目录约定，放在顶层 `skills/` 目录中，便于 agent 工具链发现和复用。
- skill 的内容要聚焦“开发期交互元素打标约束”，不能演变成通用前端代码规范全集。
- skill 既要帮助 agent 正确补标，也要避免过度打标、重复打标和破坏现有命名约定。
- skill 输出的是指导规则与工作流，不直接改动 CLI 的执行接口、运行时模型或 plan schema。

主要相关方包括：

- 在仓库中开发页面与组件的贡献者
- 使用 Codex、Copilot、Claude Code 等 agent 辅助开发的工程师
- 后续依赖稳定标识编写自动化、录制回放或验收脚本的维护者

## Goals / Non-Goals

**Goals:**

- 提供一个仓库级开发辅助 skill，专门指导 agent 为可交互元素补充稳定、可被自动化直接消费、并符合固定格式的专用 `className`。
- 为 agent 定义一套清晰的判断顺序：先识别交互元素，再检查是否已有符合约定的专用标识，如无则显式补充。
- 把“什么元素应该打标、什么元素不应打标、什么时候必须新增专用标识”沉淀为可复用规则。
- 通过渐进式披露组织内容，让主 `SKILL.md` 保持简洁，把命名规则、元素判定、示例放到引用资料中。
- 让该 skill 能与现有仓库协作规则兼容，不与现有产品使用类 skill 混淆。

**Non-Goals:**

- 不在本次变更中改造 `miniprogram-minium-cli` 的 CLI 命令或执行逻辑。
- 不把这个 skill 扩展成通用 CSS、样式组织或组件设计规范。
- 不要求为所有 DOM/小程序节点无差别打标，非交互性容器和纯展示节点不在默认覆盖范围内。
- 不在第一阶段引入自动扫描、自动重命名或 lint 插件这类强制化工程能力。

## Decisions

### 1. 新增独立 skill，而不是并入现有 `miniprogram-minium-cli` 产品使用 skill

决策：

- 新能力使用独立 skill 目录承载，而不是追加到现有 [skills/miniprogram-minium-cli/SKILL.md](/Users/bytedance/personal_workspace/miniprogram-minium-cli/skills/miniprogram-minium-cli/SKILL.md)。
- 推荐目录名采用 `skills/interactive-classname-tagging/`，以能力名而不是产品名命名。

原因：

- 现有 `miniprogram-minium-cli` skill 解决的是“如何使用 CLI 执行计划、准备运行时、分析产物”，而本次变更关注“开发期代码改动时如何打标”，两者触发条件和目标不同。
- 把两类内容混在一个 skill 中会让入口变胖，也会削弱 agent 对 skill 触发时机的判断精度。
- 独立 skill 更适合后续被其他仓库或页面开发流程复用。

备选方案：

- 直接扩展现有 `miniprogram-minium-cli` skill。
  - 优点：文件更少。
  - 缺点：产品使用指导和开发期协作指导混杂，职责边界不清。

结论：

- 采用独立 skill 目录承载开发期打标能力。

### 2. 采用“主入口 + references 渐进式披露”的结构

决策：

- 新 skill 至少包含：
  - `skills/interactive-classname-tagging/SKILL.md`
  - `skills/interactive-classname-tagging/references/when-to-tag.md`
  - `skills/interactive-classname-tagging/references/naming-convention.md`
  - `skills/interactive-classname-tagging/references/examples.md`
- 如后续需要兼容特定 agent，可补充 `agents/openai.yaml`，但首版不是必须项。

原因：

- 主 `SKILL.md` 适合只保留触发条件、总流程和 guardrails。
- “哪些元素需要打标”“如何命名”“示例前后对比”属于高细节内容，适合拆到引用资料，避免主文件持续膨胀。
- 这种结构与仓库里现有 skill 的组织方式保持一致，维护成本更低。

备选方案：

- 把所有规则都写在单个 `SKILL.md` 中。
  - 优点：文件数少。
  - 缺点：可读性和可维护性会快速下降。

结论：

- 使用渐进式披露作为首版结构。

### 3. 将打标范围限定为“默认需要被自动化稳定定位的交互节点”

决策：

- skill 明确把以下节点视为默认候选：
  - 可点击元素，如按钮、卡片入口、列表项操作区、可点图标
  - 可输入元素，如 `input`、`textarea`、表单输入容器
  - 可切换元素，如开关、tab、单选、复选、折叠面板开合控制
  - 可触发提交或导航的关键操作节点
- 以下节点默认不要求打标，除非它们承担交互语义或被测试明确依赖：
  - 纯布局容器
  - 纯文本展示节点
  - 装饰性图片、背景和非交互性包装层

原因：

- 只有把范围限定清楚，skill 才不会鼓励 agent 做“全量铺 className”这类噪声改动。
- 自动化与回放真正依赖的是“稳定交互锚点”，不是任意视觉节点。

备选方案：

- 对所有节点强制要求 `className`。
  - 优点：规则简单。
  - 缺点：代码噪声大，可维护性差，也容易掩盖真正重要的交互锚点。

结论：

- 首版采用“交互优先、展示从简”的打标范围。

### 4. agent 采用“识别 -> 校验专用标识 -> 补充”的最小增量策略

决策：

- skill 约束 agent 在改动前按以下顺序决策：
  1. 识别当前节点是否是交互元素或是否承担自动化关键路径。
  2. 检查目标节点是否已经具备符合仓库约定的专用 `className` 标识；普通业务样式类和布局类不视为已完成打标。
  3. 只要目标节点缺少专用标识，即使已经存在其他 `className`，也必须额外补充专用 `className`，并尽量落在最靠近交互语义的节点上。
  4. 若一个包装层和内部节点都可交互，优先选择测试和事件真正绑定的节点，避免无意义的双重专用打标。

原因：

- 这能避免把业务样式类误当作自动化锚点，从而提升定位稳定性。
- 同时仍然保留最小增量修改原则，避免为了打标而重写整片历史样式类。
- 对 agent 来说，明确决策顺序比只给命名规则更重要。

备选方案：

- 复用任意已有 `className`，只在完全没有类名时才补标。
  - 优点：说明更短。
  - 缺点：普通业务类名可能并不稳定，也无法显式区分“样式用途”和“自动化用途”。

结论：

- 把“显式专用标识必打”与“最小增量修改”同时作为 skill 的一等规则。

### 5. 专用标识命名固定采用 `minium-anchor-<4位hash>` 格式

决策：

- 新增的专用 `className` 必须与普通业务样式类明显区分，并固定采用 `minium-anchor-<4位hash>` 格式。
- 其中 `<4位hash>` 表示长度固定为 4 的短 hash，首版按小写十六进制处理，例如：
  - `minium-anchor-a1b2`
  - `minium-anchor-0f3c`
  - `minium-anchor-9d7e`
- 该 hash 的基础生成种子在首版固定为：`相对文件路径 + 元素树路径 + 交互类型`。
- 其中“相对文件路径”指当前页面或组件文件相对于仓库根目录的稳定路径，“元素树路径”指目标元素在当前视图树中的稳定层级路径，“交互类型”指 click、input、switch、submit 等交互类别。
- agent 应对上述种子做规范化拼接后计算 hash，并取前 4 位小写十六进制字符作为 `<4位hash>`。
- 若当前页面或组件内发生 4 位 hash 冲突，agent 必须在原始种子末尾追加递增冲突序号后重新计算，直到当前作用域内唯一。

原因：

- 固定格式可以让 agent 不必再做语义命名判断，减少命名分歧。
- `minium-anchor-` 前缀能把自动化锚点和视觉样式类清晰区分开。
- 短 hash 形式更适合批量打标，避免业务语义变化导致的频繁重命名。

备选方案：

- 使用 `qa-login-submit-button` 一类语义化命名。
  - 优点：人读起来更直观。
  - 缺点：agent 命名自由度太高，跨页面和跨人协作时一致性更差。
- 使用 `data-*` 风格单独引入新属性。
  - 优点：更贴近测试属性语义。
  - 缺点：用户当前明确希望通过 `className` 打标，且小程序生态中 `className` 更贴近现有开发习惯。

结论：

- 首版统一固定为 `minium-anchor-<4位hash>`。

### 6. skill 文档中要显式说明与自动化场景的关系，但不绑定单一执行器

决策：

- skill 要解释打标的主要消费方包括自动化测试、录制回放、问题排查和 AI 辅助修改。
- 但 skill 不应把规则写死到某一个测试框架或 CLI 命令上。

原因：

- 这样既保留当前 `miniprogram-minium-cli` 场景的直接价值，也不会把 skill 锁死为特定工具的附属物。
- 如果后续存在别的执行器或录制工具，也能继续复用同一套打标规则。

备选方案：

- 直接把命名和某个执行器强耦合。
  - 优点：短期落地更快。
  - 缺点：迁移性差，skill 复用范围受限。

结论：

- 强调自动化价值，但保持工具中立。

## Risks / Trade-offs

- [打标范围理解不一致，导致不同 agent 改动尺度差异大] -> 缓解：在 `when-to-tag.md` 中给出正反例，并明确默认候选与默认排除项。
- [4 位短 hash 在复杂页面中存在碰撞风险] -> 缓解：固定使用“相对文件路径 + 元素树路径 + 交互类型”作为基础种子，并在当前页面或组件作用域内执行唯一性检查，冲突时追加递增序号重算。
- [已有项目中存在历史命名，新增专用标识后会与旧样式类并存] -> 缓解：允许专用标识与业务类共存，但要求新规则只新增必要 token，不一次性重构历史样式类。
- [只靠 skill 约束，缺少工程化校验] -> 缓解：先通过文档和示例稳定 agent 行为，后续如有必要再评估 lint 或静态检查能力。

## Migration Plan

1. 在顶层 `skills/` 目录下新增 `interactive-classname-tagging/` skill 目录。
2. 编写主 `SKILL.md`，定义触发时机、适用范围、决策顺序和 guardrails。
3. 补充 `references/` 资料，分别覆盖打标范围、命名规范和代码示例。
4. 在相关仓库文档中补充该 skill 的定位和使用方式，说明它与现有 `miniprogram-minium-cli` skill 的职责边界。
5. 通过实际页面或组件修改案例验证：agent 能在不大面积扰动代码的前提下，为关键交互节点显式补充专用标识。
