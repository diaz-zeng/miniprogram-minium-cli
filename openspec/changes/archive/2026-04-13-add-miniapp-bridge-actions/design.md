## Context

当前 CLI 将自己定位为小程序自动化与测试的结构化执行层，而不是任意脚本入口。现有步骤类型主要服务于 UI 交互链路，无法直接表达本地存储、程序化路由、权限设置、剪贴板、反馈 UI、位置、媒体、文件与登录会话等基础能力。底层 Minium 已经提供通用 `wx` 方法调用桥，但仓库主规格也明确要求会话边界不得泄漏底层对象或任意脚本执行入口，因此这次设计的关键不在于“能不能调到”，而在于“如何把可控子集收敛成稳定的 agent-facing contract”。

## Goals / Non-Goals

**Goals:**
- 为高优先级能力提供一等结构化步骤：存储、路由、应用信息/启动信息、设置与授权、剪贴板、反馈 UI。
- 为中优先级能力提供受控结构化步骤：位置、媒体选择、文件传输、图片处理、设备交互、登录与会话、订阅消息。
- 统一同步与异步桥接动作的输入输出模型、错误分类与结果摘要。
- 在示例小程序和示例 regression plans 中提供与桥接能力对齐的演示场景与回归覆盖。
- 在仓库内 product-use skill 中补齐桥接步骤的编写、执行与结果分析指导，保持 agent-facing workflow 与主文档一致。
- 对受 `touristappid` 限制的桥接场景提供可观测的自动跳过语义。
- 保持 CLI 的验收型边界，不向 agent 暴露原始 `call_wx_method` 或任意方法名透传能力。

**Non-Goals:**
- 不在本次 change 中暴露通用 `call_wx_method` / `call_wx_method_async` 步骤。
- 不覆盖所有微信基础库 API，只纳入已确认的高优先级与中优先级集合。
- 不引入底层 Minium 对象透出、任意 JS 执行或宿主机命令执行能力。
- 不在本次 change 中实现支付、用户资料等高风险敏感能力。
- 不承诺 `touristappid` 运行态下仍然真实执行所有原生授权或受限能力。

## Decisions

### 1. 以“能力分组步骤”而不是“原始方法透传”对外建模
采用结构化步骤暴露桥接能力，例如 `storage.set`、`navigation.switchTab`、`app.getLaunchOptions`、`settings.get`、`clipboard.set`、`ui.showToast`、`location.get`、`media.chooseImage`、`file.upload`、`auth.login` 等，而不是提供 `wx.call` 一类通用入口。

这样做的原因是：
- 可以在计划阶段完成方法白名单与参数形状校验。
- 可以为每个动作给出稳定输出结构和错误码语义。
- 可以在后续替换底层实现时保持 agent 契约稳定。

备选方案是提供 `bridge.call` 与 `bridge.await`。该方案实现更快，但会破坏当前 CLI 的结构化边界，并显著增加 flaky 与不可诊断失败，因此不采用。

### 2. 先按优先级分两批接入，但放入同一个 change
本次 change 会同时声明高优先级与中优先级能力范围，便于统一评审和任务拆解；实现上按批次推进：
- 第一批：高优先级能力
- 第二批：中优先级能力

这样既能保证 change 范围完整，也能让实现期按依赖与风险递进，不必一次把所有异步能力同时落地。

### 3. 同步类能力优先使用同步 `wx` API，异步类能力提供显式等待结果
对于存储、应用信息、剪贴板等能力，优先映射到同步 `wx` API，例如 `getStorageSync`、`getLaunchOptionsSync`、`getSystemInfoSync`。对于上传下载、媒体选择、登录、订阅消息等异步能力，步骤输出需要提供结构化结果，并在必要时结合运行态等待或异步回包收敛。

备选方案是把所有桥接动作都统一抽象成异步 open/result 两步。该方案更统一，但会把简单场景复杂化，因此仅对异步能力使用。

### 4. 统一桥接动作的错误模型
桥接动作统一沿用当前执行器的结构化错误模型，但补充桥接上下文：
- `PLAN_ERROR`: 不支持的结构化步骤或参数形状非法
- `ACTION_ERROR`: 调用失败、超时、权限拒绝、前置条件不满足
- `ASSERTION_FAILED`: 仅用于断言类步骤，不混入桥接动作

错误 `details` 中需要包含：
- 结构化步骤类型
- 底层 wx 方法名
- 关键输入摘要
- 当前页面路径
- 可用时的底层 errMsg

### 5. 以 capability spec 驱动步骤命名与文档，而不是先写实现再补文档
这批能力横跨 plan schema、执行引擎、运行时适配和文档。先用 spec 锁定能力分组、场景和边界，可以避免实现时因 API 命名发散而返工。

### 6. 为桥接能力补齐仓库内示例场景与测试计划
除了主 CLI 能力本身，还需要在 `examples/demo-miniapp` 中提供对应的桥接能力演示页面、用户可见状态和场景说明，并在 `examples/demo-regression` 中提供可直接执行的 plan 样例。

这样做的原因是：
- 新能力若没有可运行的仓库内样例，后续回归验证成本会明显升高。
- 示例计划可以作为 API 文档之外的第二层契约，帮助 agent 和开发者快速理解步骤组合方式。

备选方案是仅补单元测试，不增加 demo page 与 regression plan。该方案成本较低，但无法验证真实计划编排与 run artifact 行为，因此不采用。

### 7. 对 `touristappid` 受限场景使用“检测后跳过”而不是“尝试执行后失败”
仓库已明确 `touristappid` 下部分原生授权与弹窗能力只能视为受限验证，因此这次桥接能力接入需要允许示例场景和计划显式声明“需要开发者自有 AppID”。执行时若检测到目标项目仍使用 `touristappid`，系统应将相关场景或步骤标记为跳过，并在摘要中说明原因。

备选方案是继续让这些场景在 `touristappid` 下执行并接受失败。该方案会把已知环境限制混淆成产品缺陷，也会污染 regression 结果，因此不采用。

### 8. 把 skill 同步视为同一 change 的交付项，而不是独立善后工作
桥接能力属于典型的 agent-facing capability change；如果只更新 API 文档和示例计划，而不更新仓库自带的 product-use skill，agent 仍然会基于旧指导生成过时 plan 或遗漏新步骤类型。

这样做的原因是：
- 仓库自带 skill 本身就是对 agent 暴露的使用契约入口之一。
- 把 skill 更新放进同一个 change，评审时才能同时校验“功能存在”与“agent 知道如何正确使用”。
- 这能减少后续以“补文档”名义再开零碎 follow-up change。

备选方案是等功能合入后再单独补 skill。该方案容易造成一段时间内主文档与 skill 指导不一致，因此不采用。

## Risks / Trade-offs

- [风险] 异步桥接能力的回包时序不稳定，尤其是上传下载、媒体选择与登录类接口。 → 缓解：在设计上区分同步与异步步骤，为异步步骤补充超时、结果轮询与错误摘要。
- [风险] 一次性声明的能力范围较大，落地时容易把实现批次混在一起。 → 缓解：任务按高优先级 / 中优先级分组，先完成高优先级闭环。
- [风险] 不同基础库版本下部分 `wx` 方法行为存在差异。 → 缓解：步骤文档明确前置条件，适配层对“不存在/未实现”给出结构化错误。
- [风险] 如果步骤命名过细，会造成 plan 面过于碎片化。 → 缓解：按能力域设计命名空间，在可读性与覆盖面之间取平衡。
- [风险] `touristappid` 跳过语义如果设计得过粗，可能误伤可执行场景。 → 缓解：只对显式声明需要自有 AppID 的场景或步骤启用跳过，并在摘要中记录原因。

## Migration Plan

1. 在 change 中先定义 capability spec、命名空间和能力边界。
2. 先落高优先级步骤类型、输入输出模型与基础测试。
3. 在示例小程序中补齐桥接能力对应的演示页面、场景说明和 regression plan。
4. 为依赖自有 AppID 的场景增加 `touristappid` 检测与跳过语义。
5. 在高优先级能力稳定后继续接入中优先级异步能力。
6. 更新英文/中文 API 文档与示例计划。
7. 更新仓库内 product-use skill 的引用资料，覆盖桥接步骤的编写、执行与结果分析。
8. 如后续发现单个能力域过大，再拆分为后续 change，而不回退本次结构化命名。

## Open Questions

- 中优先级异步动作是否统一要求返回 `operationId`，还是尽量在单步骤内等待完成后返回最终结果。
- 路由类动作是否继续沿用现有页面等待逻辑，还是为桥接动作单独引入可配置等待模式。
- 授权与设置类动作是否需要在 placeholder runtime 中提供更明确的仿真行为。
- 计划层面对“需要开发者自有 AppID”的标记是采用步骤级字段、计划级 metadata，还是两者同时支持。
